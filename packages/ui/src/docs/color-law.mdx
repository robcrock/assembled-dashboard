# The color law

**The palette spends its one loud color ‚Äî the Braun utility orange ‚Äî on exactly one
meaning: a broken promise.**

An orange pixel anywhere on the page IS a breach. Queue SLA breach and an agent out
of adherence (the agent-side *schedule* breach) both take it; `--status-breached`
is a semantic alias of `--sla-breach`, so "breached" and "the accent" cannot drift
apart; `--destructive` aliases it too ‚Äî a failed action and a broken promise speak
with the same voice. `SparkBars` spends it per-bar on samples past the SLA target:
same meaning, per tick.

Healthy deliberately speaks in the page's **muted ink** ‚Äî a calm floor should
*recede*, and the glyph carries the verdict (status reads by glyph shape first,
color second; `STATUS_META` is the single source for glyph + label + color, so no
surface can invent its own pairing).

## The sentiment register ‚Äî reserved, unspent

The system has two chromatic registers, and only one of them is spent:

- **The verdict register** ‚Äî the status scale. *Is a promise kept?* Healthy /
  at-risk / breached, spent on badges, dots, rows, meters. Fully allocated.
- **The annotation register** ‚Äî sentiment. *Did a number move in a welcome or an
  unwelcome direction?* `--sentiment-positive` (teal) and `--sentiment-negative`
  (red), minted with the full role anatomy (`-foreground`, `-bg`, both themes,
  AA-solved inks) ‚Äî and consumed by **nothing**.

**Red is not orange.** Orange is the alarm: a broken promise, page-level urgency,
allowed on fills, badges, and bars. Red-as-negative ‚Äî if it is ever spent ‚Äî
annotates *direction* on a figure whose verdict already lives elsewhere; it would
appear as ink on numerics only, never as a fill, badge, or alarm surface. The hues
are separated on purpose (red sits at hue 29.7, a full step from the accent's
43.6, at darker, lower-chroma ink steps), so a red annotation cannot be mistaken
for a breach pixel. Teal is quieter still ‚Äî the anchor is low-chroma by nature, a
murmur next to the alarm hues.

Why mint tokens nothing uses? So that when a consumer finally earns one ‚Äî a
domain conversation, not a CSS one ‚Äî the change is a *token pick*, not a color
invention. The law below still holds in full; amending it would be a one-token
edit instead of a per-component negotiation.

## No exceptions ‚Äî and the debt each one would create

- **Deltas are colorless, without exception.** `MetricDelta` renders muted ink and
  an explicit +/‚àí sign ‚Äî even the queue table's headroom percent. The verdict
  already lives in that row's Status badge; a second colored number would compete
  with it, and the first "small red delta" is the precedent that dissolves the law.
  The sentiment tokens do not change this ‚Äî they exist so that if the law is ever
  amended, it's amended at one token, not per-component; today no component
  consumes them.
- **Deviation bars are colorless.** Over-forecast is an *indicator* (the leading
  signal of the next breach), not a verdict ‚Äî tint it and every over-forecast bar
  becomes a false alarm that trains the eye to ignore orange.
- **No per-call color props.** `StatusBadge`, `StatusDot`, `Sparkline`, `SparkBars`
  take a `status`, never a color; the canonical classes merge *after* `className`,
  so a consumer cannot re-tint them even by accident.
- **Focus is ink, not color.** The ring token is foreground ink ‚Äî focus should be
  the sharpest thing on the page, not a fourth accent.

If a new surface seems to need a new color, it needs a new *status* in the domain
language first ‚Äî that's a model conversation, not a CSS one.
