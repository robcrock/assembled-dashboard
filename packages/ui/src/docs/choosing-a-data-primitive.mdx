# Choosing a data primitive

The decision table for anyone (or any agent) about to visualize a value. Ask what
the *question* is, not what the data looks like ‚Äî several of these accept the same
array of numbers and answer different questions with it.

| Question | Reach for | NOT for |
|---|---|---|
| How has this **moved over time** ‚Äî and did it cross a limit? | `SparkBars` (zero-baseline bars, so magnitude is honest; pass `threshold` and over-bars take the breach accent, not overridable ‚Äî omit it and the series stays neutral) | reading an exact figure at a point; a trend carries shape, and the number rides beside it |
| How **full / saturated** are we against a bound? | `Meter` (0‚Üímax fill; the attainment tile's bottom line is its consumer) | distance from a *target in the middle* ‚Äî that's DeviationBar |
| How far **over or under a baseline** are we, signed? | `DeviationBar` (colorless; the baseline dot IS the target; sign is the direction) | verdict rendering (no status tint ‚Äî verdicts live on badges beside it); showing the exact number (it renders none) |
| A **labeled number** with live-feed states? | `StatCard` (`variant` card/plain, `size` default/lg, one `feed` prop) | color-by-status cards (alarm ink is the consumer's, passed in the value node) |
| Many **entities, compared and ranked**? | `DataTable` (sortable, keyboard-navigable, expandable rows, owns feed states) | tiny fixed lists where a table's chrome outweighs three values |
| A **duration** in human terms? | `Duration` (semantic `<time>`) | wall-clock "updated Xs ago" ‚Äî that's StaleIndicator, the only wall-clock surface |
| A **signed change** vs. a target? | `MetricDelta` (colorless, +/‚àí sign, optional unit) | anything needing a verdict color or an arrow ‚Äî deliberately impossible here |
| A **verdict**? | `StatusBadge` / `StatusDot` (discriminated `status`, glyph first) | free-form colored chips ‚Äî no per-call color props exist |
| **How far is this from its target?** | `DeviationCell` (a bullet graph: measures over a signed bar whose baseline dot IS the target) | `Meter` ‚Äî that answers "how full", which is unsigned and cannot point either way |
| **May an operator change it?** | declare the key in the entity's `Setting` union, then `content.text/number/duration/enum/multiselect({ edits })` ‚Äî see [content and settings](?path=/docs/rules-content-and-settings--docs) | a `readOnly` flag; read-only is the ABSENCE of an `edits` call, and absence cannot be forgotten |

## The pattern behind the table

Every one of these is **domain-agnostic**: it takes value objects, and no entity
ever crosses the package boundary. Most are also stateless ‚Äî the exception is
`DataTable`, which owns its sort, its expansion, its selection and the one open
cell edit, deliberately: those are invariants ACROSS rows, so no row can hold
them, and a consumer that had to would be wiring with no benefit. Feed states
belong to the owners (StatCard, DataTable ‚Äî see
[feed-states](?path=/docs/rules-feed-states--docs)).
When two of them seem to fit, prefer the one whose *omissions* match your case: if
you're fighting a primitive's missing prop (a color, a number, an arrow), the
omission is usually the system telling you the meaning belongs on another surface.

Composition beats configuration: `StatCard` takes a `SparkBars` as children; the
queue table wraps `DeviationBar` in its own cell anatomy; `DeviationCell` takes
its measures as nodes. A missing feature is composed at the call site, not added
as a prop, until a *second* consumer earns it.

The table is also shorter than it was. A `Sparkline` (line trend, no threshold
concept) and a `Gauge` (one 0‚Äì100 reading as an arc) both used to sit here, and
both were deleted rather than defended: `SparkBars` already answers the trend
question ‚Äî its `threshold` is optional, so omitting it renders exactly the neutral
series `Sparkline` drew ‚Äî and no surface ever reached for the arc. **Two
primitives answering one question is worse than one**, because the table above
stops being a decision and starts being a preference. This system mints on first
use, and unmints on last.
